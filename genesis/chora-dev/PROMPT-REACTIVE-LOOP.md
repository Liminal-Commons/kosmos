## Architectural Principle — Reconciliation as Composition

The reactive system has three moments that compose into reconciliation loops:

```
mutation ──→ reflex ──→ reconciler ──→ actuality mode
             (detect)    (decide)       (act)
                                          │
             ┌────────────────────────────┘
             └── sense (feeds back as mutation)
```

- **Reflex** detects a change (trigger matches mutation)
- **Reconciler** decides what to do (compares intent with actuality via transition rules)
- **Actuality mode** executes on the substrate (manifest/sense/unmanifest)
- **Daemon** periodically senses, feeding actuality back as entity updates that close the loop

These are not three separate subsystems. They are **one compositional unit** — a reconciliation cycle. The cycle is the atom of reactive behavior. Just as an affordance is meaningless without its attainment, a reconciler is meaningless without the reflex that detects drift and the actuality mode that resolves it.

### Why this anticipates the spiral

The generative spiral (PROMPT-GENERATIVE-SPIRAL.md) identifies the reconciliation cycle as a **composite element** (Smell #4): "trigger entity + reflex entity + reconciler + actuality-mode + bonds between them. Six artifacts that must be consistent with each other."

If the reconciler engine is generic and schema-driven, the spiral can **generate** reconciliation cycles. A `generate-reconciliation-cycle` praxis would produce:
1. A reconciler entity (transition rules)
2. A trigger entity (mutation pattern)
3. A reflex entity (detect drift, invoke reconcile)
4. An actuality-mode entry (if the substrate bridge doesn't exist)
5. A daemon entity (periodic sensing)
6. Bonds wiring them together

All five artifacts follow schemas. All five can be schema-constrained in a governed inference call. But the engine must be generic first — it must read reconciler entities from the graph and execute their transition tables, not hardcode behavior per eidos.

- **No hardcoded eidos matching in reconciliation.** The reconciler engine reads `reconciler` entities and applies their `transitions` table generically. No `match eidos { "deployment" => ... }`.

- **No polling where events suffice.** Daemons sense periodically. Reflexes detect drift from sensing results. Together they replace ad-hoc polling loops.

### What's Missing — The Three Gaps

**Gap 1: Generic Reconciler Engine**

No code reads `reconciler` entities from the graph and executes their transition tables. The reconciler definitions in `genesis/dynamis/reconcilers/dynamis.yaml` are inert — they describe intent/actuality transitions but nothing interprets them. The federation reconciler in `reconciler.rs` is a bespoke implementation, not a generic engine.

What's needed: a `reconcile` operation (praxis or stoicheion) that:
1. Loads the reconciler entity for the target eidos
2. Reads the entity's `intent_field` and `actuality_field`
3. Looks up the matching transition in the `transitions` table
4. Dispatches the action (`manifest`, `unmanifest`, `sense`, or `none`)

**Gap 2: Schema-Driven Actuality Dispatch**

`host.rs:manifest()` contains a large match on eidos strings (`"deployment"`, `"task"`, `"release-artifact"`, `"dns-record"`, etc.). The generated dispatch table in `actuality_modes.rs` maps (mode, provider) -> stoicheion, but `manifest()` doesn't use it. Instead, it hardcodes which eidos gets which stoicheion.

What's needed: `manifest()` reads `actuality_mode` and `provider` from the entity data, calls `actuality_stoicheion(mode, provider, op)` from the generated dispatch table, and executes the returned stoicheion. Adding a new actuality mode requires only YAML changes, not Rust code.

**Gap 3: The Cycle Isn't Wired**

The individual pieces exist but aren't connected as a cycle:
- Daemons sense periodically (working) -> they update entity actuality fields
- Reflexes detect mutations (working) -> but no reflex triggers reconciliation on actuality drift
- Reconciler entities define transitions (defined) -> but no engine reads them
- Actuality modes dispatch (generated) -> but manifest() doesn't use the dispatch table

The cycle should be: daemon senses -> updates `actual_state` -> reflex detects `actual_state != desired_state` -> fires `reconcile` -> reconciler looks up transition -> dispatches `manifest`/`unmanifest` -> substrate changes -> next sense confirms alignment.

## Target State

### The Generic Reconciler

A new operation — callable as a praxis step or directly — that interprets reconciler entities:

```
reconcile(reconciler_id, entity_id)
  1. Load reconciler entity (target_eidos, intent_field, actuality_field, transitions)
  2. Load target entity
  3. Read intent = entity.data[intent_field]
  4. Read actual = entity.data[actuality_field]
  5. Find matching transition where transition.intent == intent AND actual IN transition.actual
  6. If action == "manifest": call manifest(entity_id)
  7. If action == "unmanifest": call unmanifest(entity_id)
  8. If action == "sense": call sense_actuality(entity_id)
  9. If action == "none": skip
```

This is pure data interpretation. No domain logic. Any topos can define a reconciler entity, and the engine executes it.

### Schema-Driven manifest/sense/unmanifest

Refactor the entry points to use the dispatch table:

```
manifest(entity_id)
  1. Load entity
  2. Read mode = entity.data.actuality_mode (or infer from eidos)
  3. Read provider = entity.data.provider (or default)
  4. stoicheion = actuality_stoicheion(mode, provider, Manifest)
  5. params = map_params_from_entity(entity, stoicheion)
  6. execute_stoicheion(stoicheion, params)
  7. Update entity.data.actual_state, manifest_handle, last_reconciled_at
```

The `actuality_mode` and `provider` fields on entity data drive dispatch. The dispatch table is already generated. Only the wiring in `manifest()`/`sense_actuality()`/`unmanifest()` needs to change.

### The Wired Cycle

For each reconcilable eidos (e.g., deployment), the cycle is:

```yaml
# 1. Daemon: periodic sensing
daemon/sense-deployments:
  praxis: dynamis/sense-all-deployments  # gathers deployments, calls sense_actuality each
  interval: 60

# 2. Reflex: detect drift (sense updates actual_state, reflex fires if it changed)
reflex/deployment-drift:
  trigger:
    event: entity_updated
    eidos: deployment
    condition: "$entity.data.actual_state != $entity.data.desired_state"
  response:
    praxis: dynamis/reconcile
    params:
      reconciler_id: reconciler/deployment
      entity_id: "$entity.id"

# 3. Reconciler: transition lookup (already defined in genesis)
reconciler/deployment:
  transitions:
    - intent: running, actual: [stopped, failed] -> manifest
    - intent: stopped, actual: running -> unmanifest
    - intent: running, actual: running -> none

# 4. Actuality mode: dispatch (already generated)
# manifest -> actuality_stoicheion("process", "local", Manifest) -> "spawn-process"
```

No new entity types. No new bond types. Just wiring existing pieces together.

## How This Enables the Spiral

When the reconciler engine is generic:

1. **Generated reconcilers work immediately.** The spiral's `generate-reconciliation-cycle` produces a reconciler entity with a transitions table. The engine reads it and executes it — no code changes needed per generated cycle.

2. **Generated actuality modes plug in.** The spiral generates an actuality-mode YAML entry. `build.rs` regenerates the dispatch table. New mode works on next build.

3. **Generated reflexes fire automatically.** The spiral produces a reflex entity with trigger bonds. The reflex registry loads it at bootstrap (or refreshes live). It fires when mutations match.

4. **Generated daemons sense automatically.** The spiral produces a daemon entity. The daemon loop discovers it and starts invoking.

5. **The composite pattern is uniform.** Every reconciliation cycle — whether hand-authored or generated — follows the same structure: reconciler + reflex + daemon + actuality-mode. The spiral generates the composite; the runtime composes it.

6. **Theoria accumulates at the cycle level.** "This transition table pattern worked well for upload artifacts" crystallizes and surfaces in future reconciliation cycle generations.

The key insight: **the reactive loop is the execution substrate for the generative spiral's output.** The spiral generates definitions (reconcilers, reflexes, daemons). The reactive loop executes them. If the loop requires code changes for each new cycle, the spiral can't work autonomously. Generic engines make the spiral productive.
